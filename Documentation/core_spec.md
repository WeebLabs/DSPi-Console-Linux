## 1. Overview

`dspi-core` is a Rust library compiled as both a **dynamic library** (`.dylib` / `.so` / `.dll`) and a **static library** (`.a` / `.lib`). It exposes a pure C ABI — every public function uses `extern "C"` calling convention and operates on C-compatible types.

The library provides:

- **USB device discovery and connection** via libusb
- **Complete protocol implementation** for the DSPi firmware (RP2040 / RP2350)
- **DSP math** for biquad coefficient calculation and frequency response curves
- **State caching** — after any command, the internal state is updated; the GUI can read it at any time
- **Hot-plug detection** via polling

The GUI never sends raw USB transfers. It calls high-level functions like `dspi_set_preamp(core, -3.5)` and the library handles encoding, transmission, and state update.

## 4. Core Lifecycle

Every interaction with the library starts by creating a core instance and ends by freeing it. The core instance is an **opaque pointer** — you cannot inspect its fields directly. Use the accessor functions described in this document.

### `dspi_core_new`

```c
struct FfiCore *dspi_core_new(void);
```

Creates and returns a new core instance. This:
- Initializes the internal device manager (no devices scanned yet)
- Sets all DSP state to defaults
- Initializes the Rust `env_logger` (first call only; logs to stderr if `RUST_LOG` env var is set)

**Returns:** Opaque pointer. Never NULL (panics on allocation failure).

**You must call `dspi_core_free` when done**, or the memory is leaked.

### `dspi_core_free`

```c
void dspi_core_free(struct FfiCore *core);
```

Destroys the core instance. Closes any active USB connection. Safe to call with NULL (no-op).

**After calling this, the pointer is invalid.** Do not use it again.

### Typical lifecycle

```c
struct FfiCore *core = dspi_core_new();

// ... use the library ...

dspi_core_free(core);
core = NULL;
```

## 5. Data Types

All types use `#[repr(C)]` in Rust, meaning they have C-compatible memory layout. The C header declares them as `typedef struct`.

### `FilterType` (enum)

```c
typedef enum FilterType {
    Flat = 0,       // Band disabled (pass-through)
    Peaking = 1,    // Parametric EQ bell
    LowShelf = 2,   // Low shelf
    HighShelf = 3,  // High shelf
    LowPass = 4,    // Low-pass (Butterworth-style)
    HighPass = 5,   // High-pass (Butterworth-style)
} FilterType;
```

`Flat` means the band is inactive. The firmware ignores bands with type=Flat.

### `FilterParams`

```c
typedef struct FilterParams {
    enum FilterType filter_type;
    float freq;   // Center/corner frequency in Hz (20–20000)
    float q;      // Quality factor (0.1–30.0; 0.707 = Butterworth)
    float gain;   // Gain in dB (-30.0 to +30.0; only used by Peaking, LowShelf, HighShelf)
} FilterParams;
```

**Defaults:** `{ Flat, 1000.0, 0.707, 0.0 }`

**Gain quantization:** The core automatically rounds gain to 0.1 dB resolution when sending to hardware. This matches the firmware's internal precision. The formula is `round(gain * 10) / 10`, with `-0.0` normalized to `0.0`.

**For LowPass and HighPass**, the `gain` field is ignored by the coefficient calculation — the filter's slope is determined entirely by `q`.

### `SystemStatus`

```c
typedef struct SystemStatus {
    float peaks[11];        // Peak levels per channel, range 0.0–1.0
    uint8_t cpu0;           // Core 0 CPU usage percentage (0–100)
    uint8_t cpu1;           // Core 1 CPU usage percentage (0–100)
    uint16_t clip_flags;    // Bitmask: bit N set = channel N is clipping
    uint8_t num_channels;   // Number of valid entries in peaks[] (7 or 11)
} SystemStatus;
```

Only `peaks[0..num_channels]` contain valid data. The rest are zero.

**Peak values** are normalized: `raw_u16 / 32767.0`. A value of 1.0 means full scale.

**Clip flags** are a bitmask. Bit 0 = channel 0, bit 1 = channel 1, etc. The firmware sets these when any sample exceeds the clipping threshold. They persist until cleared by `dspi_clear_clips`.

### `DeviceInfo`

```c
typedef struct DeviceInfo {
    uint8_t serial[64];     // NUL-terminated ASCII serial number
    uint32_t serial_len;    // Length of serial string (excluding NUL)
    uint32_t location_id;   // USB location (bus << 8 | address)
} DeviceInfo;
```

The `serial` field is a fixed-size buffer. Read `serial_len` bytes from it to get the string, or treat it as a NUL-terminated C string.

The `location_id` is derived from the USB bus number and device address. It can change if the device is plugged into a different port.

### `PlatformInfo`

```c
typedef struct PlatformInfo {
    uint8_t platform_id;         // 0 = RP2040, 1 = RP2350
    uint8_t num_channels;        // 7 (RP2040) or 11 (RP2350)
    uint8_t num_output_channels; // 5 (RP2040) or 9 (RP2350)
    uint8_t firmware_version;    // Firmware version byte
} PlatformInfo;
```

This is not directly returned by any FFI function, but its fields are available in `DspState` after `dspi_fetch_all`.

### `PresetDirectory`

```c
typedef struct PresetDirectory {
    uint16_t occupied_mask;  // Bitmask: bit N set = slot N contains a preset
    uint8_t startup_mode;    // 0 = load specified default, 1 = load last active
    uint8_t default_slot;    // Slot index to load at startup (when mode=0)
    uint8_t last_active;     // Last preset slot that was loaded
    bool include_pins;       // Whether pin config is saved with presets
} PresetDirectory;
```

There are 10 preset slots (0–9). Check `occupied_mask & (1 << slot)` to see if a slot has data.

### `DspState`

```c
typedef struct DspState {
    // Global
    float preamp_db;
    bool bypass;

    // Loudness
    bool loudness_enabled;
    float loudness_ref_spl;     // Reference SPL (dB), default 83.0
    float loudness_intensity;   // Intensity percentage (0–100), default 100.0

    // Crossfeed
    bool crossfeed_enabled;
    uint8_t crossfeed_preset;   // 0=Default, 1=Chu Moy, 2=Jan Meier
    float crossfeed_freq;       // Crossover frequency (Hz)
    float crossfeed_feed;       // Feed level (dB)
    bool crossfeed_itd;         // Interaural Time Delay enabled

    // Per-channel delays
    float channel_delays[MAX_CHANNELS];  // ms, integer values

    // Matrix mixer (2 inputs × 9 outputs)
    bool matrix_routing[2][MAX_OUTPUTS];
    float matrix_gain[2][MAX_OUTPUTS];   // dB
    bool matrix_invert[2][MAX_OUTPUTS];  // Phase invert

    // Output settings
    bool output_enabled[MAX_OUTPUTS];
    bool output_muted[MAX_OUTPUTS];
    float output_gain_db[MAX_OUTPUTS];
    float output_delay_ms[MAX_OUTPUTS];

    // Pin configuration (5 physical outputs)
    uint8_t output_pins[MAX_PHYSICAL_OUTPUTS];

    // EQ bands: [channel][band]
    struct FilterParams filters[MAX_CHANNELS][BANDS_PER_CHANNEL];

    // Channel names: [channel][32 bytes, NUL-terminated ASCII]
    uint8_t channel_names[MAX_CHANNELS][CHANNEL_NAME_LEN];

    // Platform info
    uint8_t platform_id;           // 0=RP2040, 1=RP2350
    uint8_t num_channels;          // 7 or 11
    uint8_t num_output_channels;   // 5 or 9

    // Core 1 mode
    uint8_t core1_mode;            // 0=IDLE, 1=PDM, 2=EQ_WORKER

    // Preset state
    uint16_t preset_occupied;      // Bitmask of occupied slots
    uint8_t preset_names[MAX_PRESETS][CHANNEL_NAME_LEN];
    uint8_t active_preset_slot;
    uint8_t preset_startup_mode;
    uint8_t preset_default_slot;
    bool preset_include_pins;
} DspState;
```

This is the complete state of the DSP device, cached locally. It is updated automatically by every `dspi_set_*` and `dspi_fetch_*` call. You can read it at any time via `dspi_get_state`.

## 7. Device Management

### `dspi_scan_devices`

```c
uint32_t dspi_scan_devices(
    struct FfiCore *core,
    struct DeviceInfo *out_devices,  // Array to fill
    uint32_t max_devices             // Array capacity
);
```

Scans the USB bus for all connected DSPi devices (VID `0x2E8A`, PID `0xFEAA`). Writes up to `max_devices` `DeviceInfo` structs into `out_devices`.

**Returns:** Number of devices found (may be 0).

**Example:**

```c
DeviceInfo devices[8];
uint32_t count = dspi_scan_devices(core, devices, 8);
for (uint32_t i = 0; i < count; i++) {
    printf("Device: %.*s\n", devices[i].serial_len, devices[i].serial);
}
```

### `dspi_select_device`

```c
bool dspi_select_device(struct FfiCore *core, const char *serial);
```

Opens and claims the USB device with the given serial number (NUL-terminated C string). If another device was already selected, it is disconnected first.

**Returns:** `true` if the device was opened successfully, `false` if not found or could not be opened.

**On Linux:** Automatically calls `set_auto_detach_kernel_driver(true)` to handle kernel driver conflicts.

**After connecting**, call `dspi_fetch_all` to populate the state.

### `dspi_disconnect`

```c
void dspi_disconnect(struct FfiCore *core);
```

Closes the current USB connection. The selected serial number is **retained** internally — if the same device reappears during hot-plug polling, it will be auto-reconnected.

### `dspi_is_connected`

```c
bool dspi_is_connected(const struct FfiCore *core);
```

Returns `true` if a device is currently open and connected.

## 8. State Model

The core maintains a complete cache of all device parameters in a `DspState` struct. Every `dspi_set_*` call updates the local state **and** sends the value to hardware. Every `dspi_fetch_*` call reads from hardware **and** updates the local state.

### `dspi_get_state`

```c
const struct DspState *dspi_get_state(const struct FfiCore *core);
```

Returns a **read-only pointer** to the current cached state. This pointer is valid until the next **mutable** FFI call (any `dspi_set_*`, `dspi_fetch_*`, `dspi_scan_*`, etc.).

**Usage pattern:** Call `dspi_get_state`, read the fields you need, and release the pointer before making another call. In practice with Qt, copy the fields you need into your QML-visible properties.

**Example (C++):**

```cpp
const DspState *state = dspi_get_state(core);
float preamp = state->preamp_db;
bool bypass = state->bypass;
int numChannels = state->num_channels;
// Now safe to make another call
```

### Reading channel names from state

Channel names are stored as `uint8_t[32]` NUL-terminated ASCII in `DspState.channel_names`. To use in C++:

```cpp
const DspState *state = dspi_get_state(core);
std::string name(reinterpret_cast<const char*>(state->channel_names[ch]));
```

### Reading preset names from state

Same pattern for `DspState.preset_names`:

```cpp
std::string presetName(reinterpret_cast<const char*>(state->preset_names[slot]));
```

## 9. Bulk Operations

### `dspi_fetch_all`

```c
bool dspi_fetch_all(struct FfiCore *core);
```

The **primary initialization function.** Call this after connecting to a device. It performs a single 2832-byte USB bulk transfer that retrieves every parameter at once, then additionally fetches:
- Core 1 mode
- Preset directory (occupied slots, startup mode)
- All 10 preset names
- Active preset slot

After this call, `dspi_get_state` returns fully populated state.

**Returns:** `true` on success, `false` if the device is disconnected or transfer failed.

**When to call:**
- After `dspi_select_device` succeeds
- After `dspi_load_preset` (called automatically by the library)
- After `dspi_load_params` (called automatically by the library)
- After `dspi_factory_reset` (called automatically by the library)

### `dspi_fetch_status`

```c
bool dspi_fetch_status(struct FfiCore *core, struct SystemStatus *out_status);
```

Reads real-time metering data from the device: peak levels, CPU usage, and clip flags. This is a lightweight transfer designed to be polled rapidly.

**Polling frequency:** Call this every **60 milliseconds** from a background timer to get smooth meter updates. This matches the macOS app's polling rate.

**Returns:** `true` on success. Writes the result into `*out_status`.

**Example:**

```cpp
// In a 60ms timer callback:
SystemStatus status;
if (dspi_fetch_status(core, &status)) {
    for (int i = 0; i < status.num_channels; i++) {
        updateMeter(i, status.peaks[i]);
    }
    updateCPU(status.cpu0, status.cpu1);

    // Clip detection
    for (int i = 0; i < status.num_channels; i++) {
        if (status.clip_flags & (1 << i)) {
            showClipIndicator(i);
        }
    }
}
```

## 10. EQ / Filters

The DSPi has up to 11 channels, each with 10 parametric EQ bands. Channels 0–1 are the master input (L/R), and channels 2+ are output processing channels.

### `dspi_set_filter`

```c
bool dspi_set_filter(
    struct FfiCore *core,
    uint8_t ch,        // Channel index (0–10)
    uint8_t band,      // Band index (0–9)
    struct FilterParams params
);
```

Sets all four parameters (type, frequency, Q, gain) for a single EQ band in one USB transfer.

The gain is automatically quantized to 0.1 dB before sending. The internal state is updated before the USB transfer, so the GUI can immediately reflect the change.

**Returns:** `true` on success.

**To disable a band**, set `filter_type = Flat`. The firmware skips flat bands entirely (zero CPU cost).

**Example:**

```c
FilterParams params = { .filter_type = Peaking, .freq = 1000.0f, .q = 1.41f, .gain = -3.0f };
dspi_set_filter(core, 0, 0, params);  // Channel 0, band 0
```

### Reading current filter state

After `dspi_fetch_all`, read the cached state:

```c
const DspState *state = dspi_get_state(core);
FilterParams band0 = state->filters[channel][band];
```

## 11. Global Controls (Preamp & Bypass)

```c
bool dspi_set_preamp(struct FfiCore *core, float db);
```

```c
bool dspi_set_bypass(struct FfiCore *core, bool enabled);
```

When `true`, bypasses all EQ processing on master channels (0 and 1). Outputs are unaffected. Updates `state.bypass`.

## 13. Loudness Compensation

Loudness compensation applies equal-loudness contour correction based on listening volume. It affects the master channels (0, 1).

### `dspi_set_loudness`

```c
bool dspi_set_loudness(struct FfiCore *core, bool enabled);
```

Enable or disable loudness compensation. Updates `state.loudness_enabled`.

### `dspi_set_loudness_ref`

```c
bool dspi_set_loudness_ref(struct FfiCore *core, float spl);
```

Set the reference SPL (dB) — the level at which the frequency response is considered "flat" (typically 83 dB SPL). Updates `state.loudness_ref_spl`.

### `dspi_set_loudness_intensity`

```c
bool dspi_set_loudness_intensity(struct FfiCore *core, float pct);
```

Set the compensation intensity as a percentage (0–100). At 100%, the full equal-loudness correction is applied. Updates `state.loudness_intensity`.

## 14. Headphone Crossfeed

Crossfeed blends a small amount of the left channel into the right and vice versa, simulating speaker listening on headphones.

### `dspi_set_crossfeed`

```c
bool dspi_set_crossfeed(struct FfiCore *core, bool enabled);
```

Enable or disable crossfeed. Updates `state.crossfeed_enabled`.

### `dspi_set_crossfeed_preset`

Select a crossfeed preset. Also updates the local `crossfeed_freq` and `crossfeed_feed` state to the preset's known values:

| Preset | ID | Frequency | Feed |
|--------|----|-----------|------|

| Chu Moy | 1 | 700 Hz | 6.0 dB |

### `dspi_set_crossfeed_freq`

```c
bool dspi_set_crossfeed_freq(struct FfiCore *core, float freq);
```

Set the crossfeed crossover frequency in Hz. Updates `state.crossfeed_freq`.

### `dspi_set_crossfeed_feed`

```c
bool dspi_set_crossfeed_feed(struct FfiCore *core, float feed);
```

Set the crossfeed level in dB. Updates `state.crossfeed_feed`.

### `dspi_set_crossfeed_itd`

```c
bool dspi_set_crossfeed_itd(struct FfiCore *core, bool enabled);
```

Enable or disable Interaural Time Delay simulation. Updates `state.crossfeed_itd`.

## 15. Matrix Mixer

The matrix mixer routes 2 input channels (L, R from USB) to up to 9 output channels. Each crosspoint has an enable, gain, and phase invert control.

### `dspi_set_matrix_route`

```c
bool dspi_set_matrix_route(
    struct FfiCore *core,
    uint8_t input,    // 0 = Left, 1 = Right
    uint8_t output,   // 0–8 (depends on platform)
    bool enabled,     // Route this input to this output
    float gain,       // dB attenuation/boost for this crosspoint
    bool invert       // Phase invert
);
```

Sets all three properties (enable, gain, invert) for a single matrix crosspoint in one USB transfer.

Updates `state.matrix_routing[input][output]`, `state.matrix_gain[input][output]`, `state.matrix_invert[input][output]`.

### Output channel mapping

| Output Index | RP2040 | RP2350 |
|-------------|--------|--------|
| 0, 1 | SPDIF 1 L/R | SPDIF 1 L/R |
| 2, 3 | SPDIF 2 L/R | SPDIF 2 L/R |
| 4 | PDM Sub | SPDIF 3 L/R |
| 5, 6 | — | SPDIF 3 cont / SPDIF 4 L |
| 7 | — | SPDIF 4 R |
| 8 | — | PDM Sub |

Check `state.num_output_channels` to know how many outputs exist on the connected device.

## 16. Output Controls

Each output channel (0 to `num_output_channels - 1`) has independent enable, mute, gain, and delay.

### `dspi_set_output_enable`

```c
bool dspi_set_output_enable(struct FfiCore *core, uint8_t output, bool enabled);
```

Enable or disable an output. A disabled output produces no audio and uses no CPU. Updates `state.output_enabled[output]`.

**Important:** Enabling a SPDIF output while PDM is active (or vice versa) may conflict. Check `dspi_check_core1_conflict` first. See [Core 1 Mode](#21-core-1-mode).

### `dspi_set_output_gain`

```c
bool dspi_set_output_gain(struct FfiCore *core, uint8_t output, float db);
```

Set the output gain in dB. Quantized to 0.1 dB. Updates `state.output_gain_db[output]`.

### `dspi_set_output_mute`

```c
bool dspi_set_output_mute(struct FfiCore *core, uint8_t output, bool muted);
```

Mute or unmute an output. Unlike disabling, a muted output still processes audio but silences its output. Updates `state.output_muted[output]`.

### `dspi_set_output_delay`

```c
bool dspi_set_output_delay(struct FfiCore *core, uint8_t output, float ms);
```

Set the output delay in milliseconds. Rounded to nearest integer. Updates `state.output_delay_ms[output]`.

## 17. Pin Configuration

The DSPi has 5 physical output pins (SPDIF 1–4 and PDM) that can be remapped to different RP2040 GPIO numbers. Default pins are 6, 7, 8, 9, 10.

### `dspi_set_output_pin`

```c
uint8_t dspi_set_output_pin(struct FfiCore *core, uint8_t output, uint8_t pin);
```

Change the GPIO pin for a physical output.

**Parameters:**
- `output`: Physical output index (0–4). 0–3 are SPDIF, 4 is PDM.
- `pin`: New GPIO pin number.

**Returns:** Status code:
- `PIN_CONFIG_SUCCESS` (0) — Pin changed. State updated.
- `PIN_CONFIG_INVALID_PIN` (1) — GPIO number not valid.
- `PIN_CONFIG_PIN_IN_USE` (2) — Pin already assigned to another output.
- `PIN_CONFIG_INVALID_OUTPUT` (3) — Output index out of range.
- `PIN_CONFIG_OUTPUT_ACTIVE` (4) — Cannot change PDM pin while PDM is active. Disable PDM first.
- `0xFF` — Communication error.

**Valid GPIO pins for RP2040:** 0–5, 6–11, 13–22, 26–28.

**Note:** This is an IN transfer (Device→Host) despite being a "set" operation. The firmware responds with a status code.

### `dspi_fetch_output_pin`

```c
uint8_t dspi_fetch_output_pin(struct FfiCore *core, uint8_t output);
```

Read the current GPIO pin for a physical output. Returns the pin number, or `0xFF` on error. Updates `state.output_pins[output]`.

## 19. Preset Management

The firmware supports 10 preset slots (0–9). Each preset stores the complete DSP configuration.

### `dspi_save_preset`

```c
uint8_t dspi_save_preset(struct FfiCore *core, uint8_t slot);
```

Save the current device configuration to a preset slot. Returns a preset status code (see [Constants](#6-constants)). On success (`PRESET_OK`), updates `state.active_preset_slot` and sets the slot's bit in `state.preset_occupied`.

### `dspi_load_preset`

```c
uint8_t dspi_load_preset(struct FfiCore *core, uint8_t slot);
```

Load a preset from a slot. On success, the library:
1. Updates `state.active_preset_slot`
2. Waits 10ms for firmware mute period
3. Calls `fetch_all` internally to refresh all state

**This is a blocking call** that takes at least 10ms plus USB transfer time.

### `dspi_delete_preset`

```c
uint8_t dspi_delete_preset(struct FfiCore *core, uint8_t slot);
```

Delete a preset from a slot. On success, clears the slot's bit in `state.preset_occupied`.

### `dspi_set_preset_name`

```c
bool dspi_set_preset_name(struct FfiCore *core, uint8_t slot, const char *name);
```

Set the display name for a preset slot (max 31 chars). Updates `state.preset_names[slot]`.

### `dspi_get_preset_name`

```c
bool dspi_get_preset_name(
    struct FfiCore *core,
    uint8_t slot,
    char *out_buf,
    uint32_t buf_len
);
```

Fetch the preset name from the device into `out_buf` (NUL-terminated).

### `dspi_get_preset_directory`

```c
bool dspi_get_preset_directory(struct FfiCore *core, struct PresetDirectory *out_dir);
```

Fetch the preset directory metadata: which slots are occupied, startup mode, default slot, etc. Updates the corresponding fields in `DspState`.

### `dspi_set_preset_startup`

```c
bool dspi_set_preset_startup(struct FfiCore *core, uint8_t mode, uint8_t default_slot);
```

Configure what happens at device power-on:
- `mode = 0`: Load the preset in `default_slot`
- `mode = 1`: Load the last active preset

### `dspi_set_preset_include_pins`

```c
bool dspi_set_preset_include_pins(struct FfiCore *core, bool include);
```

When `true`, saving a preset also stores the current pin configuration. When loading, the pin config is restored. When `false`, pin config is left unchanged by preset operations.

## 21. Core 1 Mode

The RP2040/RP2350 has two CPU cores. Core 1 can run in different modes, and some modes conflict with each other.

### `dspi_fetch_core1_mode`

```c
int8_t dspi_fetch_core1_mode(struct FfiCore *core);
```

Returns the current Core 1 operating mode:

| Value | Mode | Description |
|-------|------|-------------|
| 0 | IDLE | Core 1 is not running any audio task |
| 1 | PDM | Core 1 is generating PDM output (subwoofer) |
| 2 | EQ_WORKER | Core 1 is processing additional EQ channels |
| -1 | Error | Communication failure |

Updates `state.core1_mode`.

### `dspi_check_core1_conflict`

```c
int8_t dspi_check_core1_conflict(struct FfiCore *core, uint8_t output);
```

Check if enabling the given output would conflict with the current Core 1 mode.

**Returns:**
- `1` — Conflict exists (e.g., trying to enable PDM while EQ worker is active, or vice versa)
- `0` — No conflict
- `-1` — Communication error

**GUI pattern:** Before enabling an output, check for conflicts:

```c
if (dspi_check_core1_conflict(core, output_idx) == 1) {
    // Show warning: "Enabling this output requires disabling [other outputs]"
    // Let user confirm before proceeding
}
```

### PDM / EQ Worker switching

On RP2040:
- PDM output is index 4
- EQ worker outputs are indices 2–3

On RP2350:
- PDM output is index 8
- EQ worker outputs are indices 2–7

You cannot have both PDM and EQ worker outputs active simultaneously. To switch:

**Switch to PDM:**
1. Disable all EQ worker outputs (`dspi_set_output_enable(core, i, false)`)
2. Enable PDM output (`dspi_set_output_enable(core, pdm_idx, true)`)

**Switch from PDM:**
1. Disable PDM output
2. Enable the desired EQ worker output

## 22. Clip Detection

### `dspi_clear_clips`

```c
bool dspi_clear_clips(struct FfiCore *core);
```

Clear the clip flags in the firmware. After calling this, `clip_flags` in subsequent `dspi_fetch_status` calls will be 0 until new clipping occurs.

**GUI pattern:** The macOS app auto-clears clips after 10 seconds of no new clipping. Implement similar logic:

```cpp
// In your status poll handler:
uint16_t newClips = status.clip_flags & ~latchedClips;
latchedClips |= status.clip_flags;
if (newClips) {
    lastClipTime = now();
}
if (latchedClips && (now() - lastClipTime) > 10.0) {
    dspi_clear_clips(core);
    latchedClips = 0;
}
```

## 23. DSP Math (Stateless)

These functions do **not** require a device connection. They perform pure math and can be called from any thread without the core instance.

### `dspi_compute_response`

```c
float dspi_compute_response(
    const struct FilterParams *filters,
    uint32_t num_filters,
    float freq
);
```

Compute the combined frequency response magnitude in dB at a single frequency for an array of filters. Uses f64 (double) precision internally to avoid numerical artifacts with high-Q filters at low frequencies.

**Filters with `filter_type = Flat`** are skipped (contribute 0 dB).

**Returns:** Magnitude in dB. Positive = boost, negative = cut, 0.0 = unity.

### `dspi_compute_magnitude_curve`

```c
bool dspi_compute_magnitude_curve(
    const struct FilterParams *filters,
    uint32_t num_filters,
    double *out_magnitudes   // Must point to at least 201 doubles
);
```

Compute a full frequency response curve over 201 logarithmically spaced points from 10 Hz to 20 kHz.

The frequency at point `i` (0–200) is: `10 ^ (1.0 + i/200 * (log10(20000) - 1.0))`.

More precisely:
- Point 0 = 10 Hz
- Point 100 ≈ 447 Hz (geometric mean of 10 and 20000)
- Point 200 = 20,000 Hz

**Returns:** `true` on success. `false` if any pointer is NULL or `num_filters` is 0.

**GUI usage:** Call this whenever filter parameters change to update the frequency response graph.

```cpp
double magnitudes[201];
const DspState *state = dspi_get_state(core);
dspi_compute_magnitude_curve(
    state->filters[channel],
    BANDS_PER_CHANNEL,
    magnitudes
);
// magnitudes[i] is dB at the i-th log-spaced frequency
```

### Frequency axis generation

To generate the X-axis labels/positions for the magnitude curve:

```cpp
double logMin = 1.0;                   // log10(10)
double logMax = log10(20000.0);        // ≈ 4.301
for (int i = 0; i <= 200; i++) {
    double logFreq = logMin + (i / 200.0) * (logMax - logMin);
    double freq = pow(10.0, logFreq);  // Hz
    // Use freq as X-axis position for magnitudes[i]
}
```

## 24. Hot-Plug Detection

The library provides polling-based hot-plug detection. This works on all platforms (Linux, macOS, Windows).

### `dspi_set_hotplug_callback`

```c
typedef void (*DeviceEventCallback)(
    uint8_t event,           // 0 = device arrived, 1 = device departed
    const char *serial,      // NUL-terminated serial number of the device
    void *user_data          // Opaque pointer passed through from registration
);

bool dspi_set_hotplug_callback(
    struct FfiCore *core,
    DeviceEventCallback callback,
    void *user_data
);
```

Register a function to be called when devices are plugged in or unplugged. The callback fires inside `dspi_poll_hotplug` (see below), on whatever thread calls that function.

**The callback receives:**
- `event = 0` for a newly arrived device
- `event = 1` for a departed device
- `serial` is a temporary NUL-terminated string — copy it if you need to keep it

### `dspi_poll_hotplug`

```c
void dspi_poll_hotplug(struct FfiCore *core);
```

Scan the USB bus for changes since the last call. For each new or departed DSPi device, fire the registered callback. Also handles **auto-reconnect**: if the previously selected device reappears, it is automatically reopened.

**Call this every 500ms** from a timer. It is a no-op if no callback is registered (but still performs auto-reconnect).

**GUI pattern (Qt):**

```cpp
// In your bridge class constructor:
QTimer *hotplugTimer = new QTimer(this);
connect(hotplugTimer, &QTimer::timeout, [this]() {
    dspi_poll_hotplug(m_core);
});
hotplugTimer->start(500);

// Register the callback:
dspi_set_hotplug_callback(m_core, onDeviceEvent, this);

// Static callback function:
static void onDeviceEvent(uint8_t event, const char *serial, void *userData) {
    auto *bridge = static_cast<DSPiBridge*>(userData);
    QString serialStr(serial);
    // IMPORTANT: Marshal to the GUI thread!
    QMetaObject::invokeMethod(bridge, [bridge, event, serialStr]() {
        if (event == 0) {
            bridge->onDeviceArrived(serialStr);
        } else {
            bridge->onDeviceDeparted(serialStr);
        }
    }, Qt::QueuedConnection);
}
```

**Important:** The callback fires on the thread that calls `dspi_poll_hotplug`. If your GUI framework requires updates on the main thread (Qt, GTK, etc.), you must marshal the event to the GUI thread yourself.

## 25. Thread Safety

The core instance is protected by an internal **Mutex**. Every FFI function locks this mutex before operating. This means:

- **Any thread** can call any FFI function at any time.
- Calls from different threads will **serialize** (one waits for the other to finish).
- You should avoid holding `dspi_get_state()` pointers across FFI calls from another thread, since the state may be mutated.

**Recommended architecture:**

1. **GUI thread:** Reads state, updates UI, dispatches user actions.
2. **Status poll thread:** 60ms timer calling `dspi_fetch_status`.
3. **Hot-plug poll thread:** 500ms timer calling `dspi_poll_hotplug`.

All three can safely share the same core pointer. The mutex ensures correctness.

**Do not call blocking FFI functions from the GUI thread** if you need smooth 60fps rendering. Specifically, `dspi_load_preset` blocks for at least 10ms. Consider running it on a worker thread.

## 26. Error Handling Patterns

The FFI layer uses simple return-code conventions:

| Return Type | Success | Failure |
|------------|---------|---------|
| `bool` | `true` | `false` (device disconnected or transfer error) |
| `uint8_t` (status code) | See status constants | `0xFF` = communication error |
| `int8_t` (with error) | `>= 0` | `-1` = communication error |

**All failures mean one of:**
1. No device connected (`dspi_is_connected` returns `false`)
2. USB transfer timed out (1 second timeout)
3. USB transfer returned less data than expected
4. Device was physically unplugged during the transfer

**On any failure**, the core marks the device as disconnected internally. Subsequent calls will fail fast until a new device is selected.

**GUI error handling pattern:**

```cpp
if (!dspi_set_preamp(core, newValue)) {
    if (!dspi_is_connected(core)) {
        showDisconnectedUI();
    }
}
```

## 27. Platform-Specific Notes

### RP2040 vs RP2350

After `dspi_fetch_all`, check `state.platform_id` to determine the hardware platform:

| Field | RP2040 (id=0) | RP2350 (id=1) |
|-------|---------------|---------------|
| `num_channels` | 7 | 11 |
| `num_output_channels` | 5 | 9 |
| PDM output index | 4 | 8 |
| EQ worker range | outputs 2–3 | outputs 2–7 |
| Physical SPDIF outputs | 2 (SPDIF 1–2) | 4 (SPDIF 1–4) |

**Always use `state.num_channels` and `state.num_output_channels`** to size your UI dynamically. Never hard-code 7 or 11.

### Linux

- Requires **udev rules** for non-root USB access. Create `/etc/udev/rules.d/99-dspi.rules`:
  ```
  SUBSYSTEM=="usb", ATTR{idVendor}=="2e8a", ATTR{idProduct}=="feaa", MODE="0666"
  ```
  Then: `sudo udevadm control --reload-rules && sudo udevadm trigger`

- The library automatically calls `set_auto_detach_kernel_driver(true)` to handle kernel driver conflicts.

### Windows

- Requires **WinUSB driver**. Install via [Zadig](https://zadig.akeo.ie/) — select the DSPi device and install the WinUSB driver.
- libusb hotplug is **not supported** on Windows. The polling fallback (`dspi_poll_hotplug`) handles this transparently.

### macOS

- libusb works alongside IOKit. The existing SwiftUI macOS app uses IOKit directly, but the Rust core can also be used on macOS for testing.
- No special driver installation needed.

## 28. Typical GUI Application Flow

### Startup

```
1. core = dspi_core_new()
2. Register hot-plug callback:  dspi_set_hotplug_callback(core, ...)
3. Scan for devices:            dspi_scan_devices(core, ...)
4. If device found:             dspi_select_device(core, serial)
5. Fetch everything:            dspi_fetch_all(core)
6. Read state:                  state = dspi_get_state(core)
7. Populate UI from state fields
8. Start 60ms status poll timer
9. Start 500ms hot-plug poll timer
```

### User changes a parameter

```
User drags preamp slider to -3.5 dB
  → dspi_set_preamp(core, -3.5)
  → State is updated internally
  → (Optional) Read back: state = dspi_get_state(core); ui.preamp = state->preamp_db
```

For **continuous controls** (sliders being dragged), call `dspi_set_*` on every value change. The USB protocol is fire-and-forget for set operations, so there is no significant overhead.

### User edits EQ

```
User changes band 0 on channel 0 to: Peaking, 1kHz, Q=1.41, -3 dB
  → FilterParams p = { Peaking, 1000.0, 1.41, -3.0 };
  → dspi_set_filter(core, 0, 0, p);
  → Recompute magnitude curve for graph:
      dspi_compute_magnitude_curve(state->filters[0], 10, magnitudes);
  → Redraw graph
```

### User loads a preset

```
uint8_t status = dspi_load_preset(core, slot);
if (status == PRESET_OK) {
    // State is already refreshed internally
    state = dspi_get_state(core);
    refreshEntireUI(state);
}
```

### Device unplugged

```
Hot-plug callback fires with event=1
  → Show "device disconnected" in UI
  → Disable all parameter controls
  → Keep status poll running (it will return false, which is cheap)

When device plugs back in:
  → Hot-plug callback fires with event=0
  → Auto-reconnect happens automatically (if same serial)
  → dspi_fetch_all(core) to re-sync
  → Re-enable UI
```

### Shutdown

```
1. Stop all timers
2. dspi_core_free(core)
```

## 29. Source File Map

All Rust source files are in `core/src/`. Here is what each file contains and when a GUI developer would care about it:

| File | Contents | GUI relevance |
|------|----------|---------------|
| **`types.rs`** | `FilterType`, `FilterParams`, `SystemStatus`, `DeviceInfo`, `PlatformInfo`, `PresetDirectory`, and all array-size constants (`MAX_CHANNELS`, etc.) | These are the types you work with directly. All exposed in the C header. |
| **`protocol.rs`** | USB request code constants (0x42–0xA1), `BulkParams` struct and parser, packet builders, status parser, name encoding helpers. | You never call these directly, but the constants appear in the C header. Understanding the binary layout is useful for debugging. |
| **`dsp_math.rs`** | `BiquadCoeffs` struct, `calculate_coefficients()`, `response_at()`, `compute_magnitude_curve()`, `quantize_gain()`, `quantize_delay()`. Sample rate is fixed at 48,000 Hz. | The `dspi_compute_response` and `dspi_compute_magnitude_curve` FFI functions call into this. Use them for your frequency response graph. |
| **`usb.rs`** | `UsbConnection` struct wrapping `rusb::DeviceHandle`, `send_control()` and `get_control()` methods, `UsbError` enum, 1-second timeout constant. | Internal plumbing. You never interact with this directly. |
| **`device.rs`** | `DeviceManager` struct handling device enumeration by VID/PID, serial-based selection, `poll_changes()` for hot-plug, auto-reconnect logic. | Exposed via `dspi_scan_devices`, `dspi_select_device`, `dspi_disconnect`, `dspi_is_connected`, `dspi_poll_hotplug`. |
| **`state.rs`** | `DspState` struct with all cached parameters, `apply_bulk_params()` method, `platform_name()` and `pdm_output_index()` helpers. | Exposed via `dspi_get_state`. This is how you read all device state. |
| **`commands.rs`** | All `set_*` and `fetch_*` command methods on `DspiCore`: EQ, preamp, bypass, delay, loudness (3 params), crossfeed (5 params), matrix route, output enable/gain/mute/delay, core1 mode/conflict, pin config, channel names, flash save/load/reset, clip clear. | Every `dspi_set_*` and `dspi_fetch_*` FFI function calls into here. |
| **`preset.rs`** | Preset `save_preset`, `load_preset`, `delete_preset`, name get/set, directory, startup mode, include-pins, active preset. Includes the 10ms sleep after load. | Exposed via `dspi_save_preset`, `dspi_load_preset`, etc. |
| **`lib.rs`** | `DspiCore` struct definition, `FfiCore` mutex wrapper, all 40+ `extern "C"` FFI functions, hot-plug callback type, `with_core` / `with_core_const` helpers. | This is the entire public API surface. Every function listed in this document is defined here. |
